Token Sort Ratio handles word-order variations that defeat character-level algorithms entirely. The three-algorithm composite produces a score that is more robust than any individual algorithm alone."

**Algorithm Details** — three cards (rounded-lg border border-border p-5 space-y-2) for each:

Card 1 — Jaro-Winkler (JW × 0.6):
- Heading: "Jaro-Winkler" with badge "Weight: 60%"
- Body: "Measures character-level similarity with extra credit for matching prefix characters. Produces a score from 0 (completely different) to 1 (identical). Highly effective for short strings — personal names, entity names — and tolerates transposition errors, abbreviations, and transliteration variants. This is the highest-weighted algorithm because name screening is inherently a short-string problem."
- When it leads: "Jaro-Winkler wins when no other algorithm produces a stronger signal. It is the default algorithm for Latin-script names without word-order issues."

Card 2 — Double Metaphone (DM × 0.25):
- Heading: "Double Metaphone" with badge "Weight: 25%"
- Body: "Encodes a name as one or two phonetic codes representing how it sounds rather than how it is spelled. If any phonetic code from the input name matches any phonetic code from the SDN entry, the algorithm contributes a 1.0 bonus to the composite score. This catches pronunciations that look completely different in writing: OSAMA and USAMAH produce the same phonetic code."
- Limitation callout: "Double Metaphone cannot encode CJK or Arabic-script names — it returns empty codes for those inputs. Two empty-code inputs do not produce a false-positive match (the tool guards against this explicitly)."
- When it leads: "Double Metaphone wins (overrides other scores) when phonetic codes overlap — signaling that two names sound identical regardless of spelling."

Card 3 — Token Sort Ratio (TSR × 0.15):
- Heading: "Token Sort Ratio" with badge "Weight: 15%"
- Body: "Splits each name on whitespace, sorts the tokens alphabetically, joins them back into a string, then computes Jaro-Winkler on the sorted result. This makes the score word-order invariant. HASSAN ALI and ALI HASSAN produce identical sorted strings and therefore score 1.0 against each other. Particularly important for Arabic names, where given name and family name order varies by country and data-entry convention."
- When it leads: "Token Sort Ratio wins when its sorted-string JW score beats the raw JW score — indicating the match is primarily a word-order issue rather than character-level similarity."

**Composite Formula**: Subheading. Formula display block (rounded-lg bg-muted/50 p-4 font-mono text-sm text-center): `composite = (JW × 0.6) + (DM_bonus × 0.25) + (TSR × 0.15)`. Then brief explanation: "The composite score is always between 0 and 1. The weights reflect empirical performance on the 285-entry synthetic dataset: JW carries the most weight because it is the most reliable for the range of name patterns in the dataset; DM and TSR provide targeted bonuses for their specific strengths."

**The Five Risk Tiers**: Subheading. Intro sentence: "Every match result is assigned to one of five risk tiers based on the composite score." Then a tier table — use a div-based layout (not an HTML table, to avoid width issues with virtualizer — use a list of tier rows styled consistently):

Five tier rows, each as a flex row with: color badge, tier name, threshold, compliance meaning:
- EXACT (bg-red-600 text-white badge): composite ≥ 0.97 — "Automatic block recommended. Score is consistent with an identical or near-identical match. Manual review required before any transaction proceeds."
- HIGH (bg-orange-500 text-white badge): composite ≥ 0.90 — "Strong hit — high confidence of a true positive. Escalate to compliance officer immediately."
- MEDIUM (bg-yellow-500 text-crowe-indigo-dark badge): composite ≥ 0.80 — "Probable hit — moderate confidence. Review match context and additional identifying information."
- LOW (bg-blue-500 text-white badge): composite ≥ 0.70 — "Possible hit — low confidence. Document review and rationale for proceeding."

The rationale: short names have less distinguishing information, so a 0.82 composite score on a 5-character name is more likely to represent a genuine match than the same score on a 15-character name. **Unicode Normalization**: Subheading. "The scoring engine normalizes every name before any algorithm runs. This pipeline ensures that a name entered with a Cyrillic О instead of a Latin O scores identically to the all-Latin version — catching a common document-scanning obfuscation pattern."
