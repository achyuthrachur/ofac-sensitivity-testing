## Summary

Phase 17.1 is a pure content-and-layout phase. No new engine code. The deliverable is a `/guide` route rendered as a Next.js App Router Server Component with a sticky sidebar for anchor navigation and four content sections covering the full tool.

The key technical question — "how do we get scrollspy (sidebar highlights active section) in a Server Component?" — resolves cleanly: the sidebar can use simple anchor `<a href="#section-id">` links with no JavaScript for basic navigation. Scrollspy requires a small `'use client'` shell (an IntersectionObserver wrapper), but this is optional per discretion area. The page itself remains a Server Component; only the sidebar highlight logic needs a client boundary.

The key layout question — sticky sidebar in Tailwind v4 — is straightforward CSS: `position: sticky; top: 1.5rem` on the sidebar div, with the content column taking flex-1. No JavaScript needed. Tailwind utility classes `sticky top-6` plus a flex row container achieve this.

The key integration question — where does the `/guide` link go? — is fully answered by reading the source. The root `layout.tsx` header is the right place (it wraps every page including `/guide`). The `HeroSection.tsx` CTA area is where a secondary "Read the guide" link belongs. Both modifications are minimal.

**Primary recommendation:** Create `src/app/guide/page.tsx` as a pure Server Component. Left sidebar 220px fixed width with `sticky top-6`. Content area `flex-1`. Sidebar uses plain `<a href="#...">` anchors. Optionally add a thin `'use client'` `GuideSidebar.tsx` shell for IntersectionObserver-based active section highlighting — isolated so the page tree remains server-rendered.

### Pattern 3: Optional Scrollspy — Thin 'use client' Shell

**What:** A `'use client'` component that uses `IntersectionObserver` to detect which section is visible and highlight the corresponding sidebar link. The sections themselves remain Server Components.

**When to use:** If discretion is exercised to add active-section highlighting.

**Example:**

```tsx
// src/app/guide/_components/GuideSidebar.tsx
'use client';
import { useEffect, useState } from 'react';

const SECTIONS = [
  { id: 'ofac-context', label: 'OFAC Context' },
  { id: 'sensitivity-test', label: 'Sensitivity Test' },
  { id: 'screening-mode', label: 'Screening Mode' },
  { id: 'scoring-engine', label: 'Scoring Engine' },
];

export function GuideSidebar() {
  const [activeId, setActiveId] = useState<string>(SECTIONS[0].id);

  useEffect(() => {
    const observers = SECTIONS.map(({ id }) => {
      const el = document.getElementById(id);
      if (!el) return null;
      const obs = new IntersectionObserver(
        ([entry]) => { if (entry.isIntersecting) setActiveId(id); },
        { rootMargin: '-20% 0px -70% 0px' }
      );
      obs.observe(el);
      return obs;
    });
    return () => observers.forEach(o => o?.disconnect());
  }, []);

  return (
    <nav className="sticky top-6 space-y-1">
      {SECTIONS.map(({ id, label }) => (
        <a
          key={id}
          href={`#${id}`}
          className={`block text-sm py-1 px-2 rounded transition-colors ${
            activeId === id
              ? 'bg-crowe-indigo-dark text-white font-semibold'
              : 'text-muted-foreground hover:text-foreground hover:bg-muted'
          }`}
        >
          {label}
        </a>
      ))}
    </nav>
  );
}
```

### Anti-Patterns to Avoid

- **Making guide/page.tsx a Client Component:** The entire page is static content. `'use client'` on page.tsx forces client-side rendering of all section content. Only GuideSidebar.tsx (optional scrollspy) needs the directive.
- **Importing scorer.ts or runTest.ts in the guide:** Guide content is prose, not live computation. Never import from the isolated v1.0 files for documentation purposes — read constants from screening.ts directly if needed (TIER_THRESHOLDS, COST_OF_MISS_MULTIPLIER).
- **Duplicating EngineExplanationPanel content:** The EngineExplanationPanel lives in the tool at the "Engine Docs" tab. The guide should be a richer, onboarding-oriented version — not a copy. Do not import or re-export EngineExplanationPanel into the guide page.
- **Using CSS modules or inline styles for the sidebar sticky:** Tailwind v4 `sticky top-6` is sufficient. Using a CSS module conflicts with the Crowe token system.
- **Adding 'use client' to section components:** Section components are pure prose markup. They do not need state or effects.

## Common Pitfalls

### Pitfall 1: Sticky Sidebar Not Sticking

**What goes wrong:** The sidebar does not stick while scrolling — it scrolls off-screen with the content.

**Why it happens:** `position: sticky` requires the parent container to have a defined height and `overflow` must NOT be set to `hidden`, `auto`, or `scroll` on any ancestor element. If the two-column flex container has `overflow-hidden` anywhere in the ancestor chain, sticky breaks silently.

**How to avoid:** Use `flex` on the outer container with no `overflow` restriction. The outer div (`max-w-6xl ... flex gap-10`) must not have `overflow-hidden`. The sidebar's `sticky top-6` applies to the `<aside>` element itself.

**Warning signs:** Sidebar disappears when scrolling past the first viewport. Check ancestor elements for any `overflow-hidden` class.

### Pitfall 2: Anchor Link Scroll Offset Hidden Under Sticky Header

**What goes wrong:** Clicking a sidebar anchor scrolls the section heading exactly to the top of the viewport — but the root layout header (`bg-crowe-indigo-dark`, ~48px tall) covers it.

**Why it happens:** The root layout header is `position: fixed` or `position: sticky`. When the browser scrolls to `#section-id`, it aligns the element top with the viewport top, not below the header.

**How to avoid:** Add `scroll-margin-top` to each section element. Tailwind utility `scroll-mt-20` (80px) provides enough clearance for the header. Apply to every `<section id="...">` element.

**Warning signs:** After clicking a nav link, the section heading is hidden behind the header.

### Pitfall 3: HeroAnimationShell Conflict When Adding Secondary CTA

**What goes wrong:** Adding a new link inside HeroSection.tsx breaks the Phase 13 CTA glow animation.

**Why it happens:** HeroAnimationShell uses `createScope` targeting `.cta-button` className. The glow animation uses `animate('.cta-button', ...)`. A new sibling link does NOT have the `.cta-button` class, so the scope is unaffected — but if the new link accidentally receives `.cta-button`, it gets animated unintentionally.

**How to avoid:** Do not add `.cta-button` className to the secondary link. The animation scopes to exactly one element. Review HeroAnimationShell.tsx before adding to HeroSection.

**Warning signs:** CTA glow animation fires on two elements, or the secondary link unexpectedly glows on page load.

### Pitfall 4: Root Layout Header Has No `<nav>` — Adding Links Breaks the Layout Balance

**What goes wrong:** The current header is `flex items-center justify-between` with only two children (Crowe span, OFAC text span). Adding a `<Link href="/guide">` between them or after them without restructuring will shift the layout.

**Why it happens:** `justify-between` with two items = left/right ends. Three items = three positions. The visual balance changes.

**How to avoid:** Restructure the header to keep "Crowe" on the left and group nav links + title on the right. See Pattern 4 above. Alternatively, add the guide link as a right-side cluster alongside the existing "OFAC Sensitivity Testing" text.

**Warning signs:** Header looks unbalanced or the "Crowe" branding shifts off-center.

### Pitfall 5: Importing TIER_THRESHOLDS or COST_OF_MISS_MULTIPLIER Creates v3.0 Type Coupling

**What goes wrong:** Importing from `src/types/screening.ts` in the guide page is technically allowed (it is not in the isolation list), but it creates a dependency from a documentation page into the v3.0 type layer.

**Why it happens:** Developer imports constants for documentation accuracy.

**How to avoid:** Hardcode the values in the guide prose. The guide is documentation — it should express values as human-readable text ("EXACT tier: 0.97 or above") rather than importing TypeScript constants. This also makes the guide readable without understanding the codebase.

**Warning signs:** Guide page imports from `@/types/screening` — this is a smell for a documentation page.
