# Domain Pitfalls

**Domain:** Next.js form-driven string transformation demo tool (OFAC sensitivity testing)
**Researched:** 2026-03-03
**Confidence:** MEDIUM — training data (Aug 2025 cutoff). Core Vercel limits and Next.js App Router behavior are stable.

---

## Critical Pitfalls

---

### Pitfall 1: Vercel Serverless Function Timeout on Large Computations

**What goes wrong:** The string transformation engine processes 500 names × N degradation rules server-side. At scale (10+ active rules, complex Unicode normalization), processing exceeds Vercel's function timeout: 10s Hobby, 60s Pro, 300s Enterprise. Client sees 504 Gateway Timeout with no partial results.

**Warning signs:** Local dev is instant but staging/prod times out; Vercel logs show `FUNCTION_INVOCATION_TIMEOUT`.

**Prevention:**
- Benchmark worst-case input (500 names, all rules active) and assert < 5 seconds.
- If timeout risk is real, move transformation logic to the client side — 500 names × 20 rules is ~10,000 string operations, trivially fast in the browser (<200ms).
- Alternatively, use streaming responses (TransformStream) so partial results appear progressively.

**Phase to address:** Core transformation engine phase, before Vercel deployment.

---

### Pitfall 2: Unicode/Script Normalization Silently Corrupting Non-Latin Names

**What goes wrong:** Degradation rules designed for ASCII are applied blindly to Arabic, Chinese, or Cyrillic strings. Arabic names lose joining characters. Chinese characters are mangled by Latin-targeting regex. Russian names through ASCII phonetic rules produce garbage or empty strings.

**Warning signs:** Transformed Arabic or Chinese names appear as empty strings; match scores cluster at 0% or 100%; unit tests only cover ASCII input.

**Prevention:**
- Classify each name by script at ingestion using Unicode ranges (`\u0600-\u06FF` Arabic, `\u4E00-\u9FFF` CJK, `\u0400-\u04FF` Cyrillic).
- Each rule declares `applicableScripts`. Rules not defined for a script are no-ops — never silently applied.
- Create test fixtures with at least 5 names per supported script; assert transformation outputs per rule.
- Use `Intl.Segmenter` for locale-aware operations rather than manual regex.

**Phase to address:** String transformation engine phase, before synthetic dataset integration.

---

### Pitfall 3: React Re-render Cascade on Form State with Hundreds of Checkboxes

**What goes wrong:** All checkbox state in a single top-level `useState`. Every checkbox change re-renders the entire tree including the results table. At 2,000+ result rows, each re-render takes 100–300ms — form feels broken.

**Warning signs:** React DevTools Profiler shows results table re-rendering on checkbox clicks; >100ms commit time on checkbox interaction.

**Prevention:**
- Isolate form state from results state completely. Different `useState`/`useReducer` atoms.
- Virtualize results table from day 1 with `@tanstack/react-virtual`.
- Memoize results component with `React.memo`.
- Use `useTransition` for processing (marks as non-urgent, keeps UI responsive).

**Phase to address:** UI component architecture, before populating with real data.

### Pitfall 4: CSV Download Generating Corrupt Files for Non-Latin Scripts in Excel

**What goes wrong:** CSV generation without UTF-8 BOM. Excel fails to decode UTF-8 CSV, displaying Arabic/Chinese as mojibake (`â€˜` etc.). Client opens the download, sees garbage, concludes the tool is broken.

**Warning signs:** Non-Latin names display as mojibake in Excel; CSV does not start with `EF BB BF` bytes.

**Prevention:**
- Always prepend UTF-8 BOM (`\uFEFF`) to CSV blob.
- Use `type: 'text/csv;charset=utf-8;'` for the Blob.
- Double-quote all values; escape internal quotes.
- **Validate by opening in actual Excel, not just a text editor** — this is an acceptance criterion.

**Phase to address:** CSV export phase.

---

### Pitfall 5: Synthetic Dataset Names That Are Linguistically Invalid

**What goes wrong:** Synthetic names are generated by randomly combining components without cultural knowledge. Results are technically in the right script but linguistically nonsensical. A compliance professional recognizes them as fake immediately.

**Warning signs:** Name combinations impossible in source culture; every Arabic name is exactly 3 tokens; no variation in name length.

**Prevention:**
- Source name components from the public OFAC SDN list and academic name corpora.
- Enforce naming conventions per script: Arabic (ism + nasab/bin), Chinese (surname-first, 1–2 char given), Russian (first + patronymic + surname), Latin (Western European first + last).
- Include a small number of actual OFAC SDN list entries to anchor the synthetic set.
- Have dataset reviewed by someone with compliance domain knowledge.

**Phase to address:** Synthetic dataset creation, before the transformation engine is built against it.

---

### Pitfall 6: No Loading/Progress Feedback During Processing

**What goes wrong:** Processing takes 1–5 seconds with no feedback. Users click "Run Test," see nothing, click again, trigger duplicate processing, get doubled or corrupted results.

**Warning signs:** Submit button stays active during processing; no visual change for 1+ seconds after click.

**Prevention:**
- Implement processing state machine (`idle | processing | complete | error`) from first form integration — not a polish task.
- Disable submit button during processing.
- Use `useTransition`; display real progress if possible ("Analyzing X of 500...").

**Phase to address:** Form + processing integration phase.

---

### Pitfall 7: Degradation Rule Order Dependency Producing Non-Deterministic Results

**What goes wrong:** Rules applied sequentially in order determined by checkbox render order or object key iteration — neither stable. Rule A removes diacritics; Rule B applies phonetic substitution that depends on diacritics being present. Same parameter set on different occasions produces different results.

**Warning signs:** Rules stored in a plain JS object; no determinism test exists.

**Prevention:**
- Define an explicit `CANONICAL_ORDER` constant (ordered array, not object).
- Apply rules in canonical order regardless of which are active; inactive rules are skipped.
- Write a determinism test: given input X with rules A+B active, output is always Y regardless of toggle order.

**Phase to address:** Transformation engine design, before any rule implementations are written.

## Moderate Pitfalls

---

### Pitfall 8: Results Table Performance Degrading on Sort/Filter

**What goes wrong:** Client-side `.sort()` / `.filter()` on 5,000+ rows are synchronous and block the main thread 200–500ms. Table freezes on every sort click or filter keystroke.

**Prevention:** `useMemo` for sort/filter; virtualization for sorted views; debounce filter inputs 150ms.

**Phase to address:** Results table component phase.

---

### Pitfall 9: Form State Not Preserved Across Processing Runs

**What goes wrong:** Form state resets on submission. User tweaks one parameter and re-runs but all 20+ checkboxes return to defaults.

**Prevention:** Never reset form state on submission — processing results in separate state. Persist form state to `localStorage` for demo resilience across page refreshes.

**Phase to address:** Form state management phase.

---

### Pitfall 10: Scoring Functions Returning NaN for Edge Cases

**What goes wrong:** Similarity scoring not validated against edge cases: empty string after transformation, single-character names, identical before/after transform. Division by zero or undefined comparisons produce `NaN` that silently propagates.

**Prevention:** Guard clauses in all scoring functions for empty strings, identical inputs, single-character inputs. Unit tests for exact edge cases before UI integration.

**Phase to address:** Scoring implementation phase.

---

### Pitfall 11: Hardcoded Thresholds That Look Arbitrary

**What goes wrong:** Score thresholds (e.g., "match" > 85%) hardcoded. Compliance professionals ask "why 85%?" The answer "it was the default" damages credibility.

**Prevention:** Expose thresholds as configurable form parameters with documented rationale in tooltips.

**Phase to address:** Form parameters phase.

---

### Pitfall 12: No Error Boundary Around Results Table

**What goes wrong:** A single malformed name crashes results rendering → white screen for entire page.

**Prevention:** Wrap results table in a React `ErrorBoundary` with a graceful fallback message.

**Phase to address:** Results component phase.

---

### Pitfall 13: CSV Column Headers Not Matching Displayed Column Names

**What goes wrong:** UI shows "Original Name" but CSV exports `original_name`. Client is confused.

**Prevention:** Drive both UI column labels and CSV headers from a single `COLUMNS` constant.

**Phase to address:** CSV export phase.

## Phase-Specific Warning Summary

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|---------------|------------|
| Transformation engine design | Rule order dependency (P7) | Define canonical order constant before writing any rules |
| Multi-script name support | Unicode corruption (P2) | Script classification + per-rule script declarations |
| Synthetic dataset generation | Linguistically invalid names (P5) | Reference real OFAC SDN structure; domain review |
| Server/client processing split | Vercel timeout (P1) | Benchmark worst case; consider client-side computation |
| Form + results integration | Re-render cascade (P3) | Separate state, virtualize, memoize |
| Processing UX | No loading feedback (P6) | State machine first, not last |
| CSV export | Corrupt non-Latin in Excel (P4) | UTF-8 BOM + test in actual Excel |
| Scoring algorithm | NaN propagation (P10) | Guard clauses + edge-case unit tests |
